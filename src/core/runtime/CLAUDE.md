- Any new path that boots a pi process (root or child) **must** set `CRTR_FRONT_DOOR=1` (via `FRONT_DOOR_ENV`) in the child's env — omitting it bypasses the recursion guard in `front-door.ts` and fork-bombs pi. See `bootRoot` and `spawnChild` in `spawn.ts` for the two existing call sites.
- **Flip status BEFORE killing a tmux window** — the daemon only revives `active|idle` nodes, so marking `done`/`canceled` first closes the race where the daemon sees a window-gone live node and revives or kills it, overwriting your transition.
- Any polymorph path (promote, demote, reshape) must rewrite `LaunchSpec` via `buildLaunchSpec` — it is the canonical revive recipe. Skipping it means the node silently revives as its old persona after the next refresh.
- `wakeOrigin` is in-memory only — never written to the stored recipe (the daemon spreads it in at fire time). Two injection seams only, both set ONLY by the daemon: `spawn.ts` (wake-born kickoff) and `kickoff.ts` `buildReviveKickoff`. `noted`/`deadline` wakes self-mark via their inbox label instead, not the `<crtr-wake>` block.
